What is DockUp?

DockUp is a minimal, zero-dependency PaaS that automatically deploys your Docker Compose applications from GitHub to your VPS. No complex setup, no bloated dependencies‚Äîjust push to GitHub and deploy.

Quick Start

0Install DockUp Globally (Recommended)

Install DockUp as a global command so you can use it from any project:

Installation Command
curl -fsSL https://raw.githubusercontent.com/Moe1211/dockup/main/install-global.sh | bash
üí° Note: This installs DockUp system-wide. After installation, you can use dockup from any directory, just like git.
‚ú® New Feature: Run dockup with no arguments to enter interactive mode! Perfect for guided deployments.

1Configure GitHub App (One-time)

Before deploying, you need to set up a GitHub App for repository access:

Configure GitHub App
dockup user@vps-ip configure-github-app
üí° Note: The DockUp GitHub App uses App ID 2330335. Installation ID can be auto-detected if GitHub CLI is configured. See GITHUB_APP_SETUP.md for detailed setup instructions.

2Deploy Your App

Navigate to your project and deploy in one command:

Deploy Command
cd my-app dockup user@vps-ip deploy

This single command will:

  • Check if DockUp is installed on VPS (setup if needed)
  • Clone your repository on the VPS
  • Register the app in DockUp
  • Trigger an immediate build and deploy
‚úÖ That's it! Your app is now deployed and will auto-update on every push to your configured branch.

3Auto-Deploy on Push

Just push to your configured branch and DockUp handles the rest:

Git Push
git push origin main

The agent will automatically:

  • Validate the webhook signature
  • Pull the latest code
  • Rebuild Docker images
  • Restart containers

Prerequisites

  • A Linux VPS (Ubuntu/Debian recommended)
  • SSH access to your VPS
  • A Git repository (GitHub, GitLab, etc.)
  • GitHub App configured (see GITHUB_APP_SETUP.md)
  • gh CLI (optional, for automatic webhook setup)

How It Works

DockUp consists of two components:

  1. Agent (Server-side): A Go binary running as a systemd service that listens for webhooks
  2. CLI (Client-side): A bash script that provisions repos via SSH

When you push to GitHub, a webhook triggers the agent, which then:

  1. Validates the webhook signature
  2. Pulls the latest code using GitHub App authentication
  3. Runs docker compose build
  4. Runs docker compose up -d

Command Reference

dockup deploy (Recommended)

Use when: You want to deploy your app in one command

What it does: Checks if DockUp is installed, registers your repository, and triggers immediate build and deploy

Example
dockup user@vps-ip deploy dockup user@vps-ip deploy --rebuild

dockup setup

Use when: First time setting up a new VPS

What it does: Installs DockUp agent, Docker, and dependencies on your VPS

Example
dockup user@vps-ip setup
Note: Usually not needed - deploy handles this automatically

dockup disconnect

Use when: You want to stop auto-deployments but keep the app running

Example
dockup user@vps-ip disconnect dockup user@vps-ip disconnect my-app

dockup remove

Use when: You want to completely remove an app from your VPS

Example
dockup user@vps-ip remove dockup user@vps-ip remove my-app
‚ö†Ô∏è Warning: This permanently deletes the app directory and all its data

dockup list

Use when: You want to see all apps registered on your VPS

Example
dockup user@vps-ip list

Shows app name, path, branch, compose file, directory status, and container status.

dockup version

Use when: Check your installed version and see if updates are available

Example
dockup version

Automatically checks for updates and shows comparison. Can be disabled with DOCKUP_SKIP_UPDATE_CHECK environment variable.

dockup (Interactive Mode) ‚ú® NEW

Use when: You want a guided, menu-driven experience

Example
dockup

Run dockup with no arguments to enter interactive mode. You'll see a numbered menu where you can:

  • Select commands from a menu
  • Enter remote host information when prompted
  • Select apps from a list (for disconnect/remove commands)
  • Get help and version information
‚úÖ Backward Compatible: All existing command-line usage still works. Interactive mode only activates when no arguments are provided.

Manual Deployment

You can trigger a manual deployment via HTTP:

Manual Deployment
curl -H "Authorization: Bearer YOUR_SECRET" \ http://vps-ip:8080/webhook/manual?app=my-app

Configuration

The agent reads from /etc/dockup/registry.json on the VPS:

Registry Configuration
{ "my-app": { "path": "/opt/dockup/apps/my-app", "branch": "main", "secret": "webhook-secret-key", "compose_file": "docker-compose.prod.yml" } }

Repository-Specific Configuration

Default Structure (No Config File Required)

By default, DockUp works with a simple repository structure. No configuration file is needed if you follow the standard setup:

Default Repository Structure
my-app/ ‚îú‚îÄ‚îÄ docker-compose.yml # Required: Docker Compose configuration ‚îú‚îÄ‚îÄ Dockerfile # Optional: For building custom images ‚îî‚îÄ‚îÄ ... # Your application files

When no config.dockup.yml is found, DockUp will:

  • Look for docker-compose.yml in the repository root
  • Use docker compose build --pull (builds from Dockerfile if present, or uses pre-built images)
  • Use docker compose up -d --remove-orphans to start containers
‚úÖ Works Out of the Box: If your repository has a docker-compose.yml file, DockUp will deploy it automatically without any additional configuration.

For advanced customization, you can optionally add a config.dockup.yml file to your repository root. This file will be automatically detected and used during deployment, allowing you to customize build commands, hooks, environment variables, and more.

config.dockup.yml Schema

config.dockup.yml Schema
# DockUp Configuration File # Place this file in your repository root # Docker Compose configuration compose: file: "docker-compose.prod.yml" # Optional, defaults to docker-compose.yml build_args: # Build arguments for docker compose build - "NODE_ENV=production" - "API_URL=https://api.example.com" # Build hooks (optional) hooks: pre_build: # Commands to run before docker compose build - "npm install" - "npm run build" post_build: # Commands to run after docker compose build - "npm run test" pre_deploy: # Commands to run before docker compose up - "echo 'Starting deployment...'" post_deploy: # Commands to run after docker compose up - "curl -X POST https://hooks.slack.com/..." # Environment variables (injected into containers) environment: NODE_ENV: "production" DATABASE_URL: "${DATABASE_URL}" # Can reference VPS environment variables API_KEY: "${API_KEY}" # Health checks health_check: enabled: true endpoint: "/health" timeout: 30 interval: 60 retries: 3 # Deployment strategy deployment: strategy: "rolling" # Options: rolling, blue-green, immediate max_parallel: 2 # For rolling deployments # Resource limits (per service) resources: web: memory: "512m" cpus: "0.5" db: memory: "1g" cpus: "1.0" # Notifications notifications: webhooks: - url: "https://hooks.slack.com/services/..." on_success: true on_failure: true - url: "https://discord.com/api/webhooks/..." on_failure: true # Rollback configuration rollback: enabled: true keep_releases: 5 # Number of previous releases to keep

Advanced Example

üìù Note: The config.dockup.yml file is completely optional. If not present, DockUp will use the default behavior described above (expects docker-compose.yml and optionally a Dockerfile). Configuration in /etc/dockup/registry.json takes precedence for system-level settings (path, branch, secret, compose_file).
